# 老人陪伴助手 PRD

## 1. 产品概述
- 产品名称：老人陪伴助手
- 当前版本：`1.3.0`
- 产品形态：手机 Web 应用
- 核心交互：语音为主，文字为辅
- 当前定位：面向独居、空巢、慢病管理和需要家属协同照护的老人，提供“陪伴 + 提醒 + 应急 + 家属联动 + 情绪报平安 + 家庭记忆沉淀 + 防诈骗守护”闭环
- 产品目标：让老人“开口就能用”，让家属“远程看得见、关键时刻收得到”

## 2. 用户角色

### 2.1 老人用户
- 年龄段：60 岁以上
- 设备：智能手机
- 使用偏好：大字、大按钮、少步骤、语音优先
- 典型痛点：
  - 不愿意学习复杂 App
  - 不爱打字
  - 容易忘记吃药、喝水、复诊
  - 身体不适时容易慌乱

### 2.2 家属用户
- 身份：子女、配偶、照护者
- 核心诉求：
  - 知道老人近期有没有互动、异常、提醒
  - 希望紧急情况下能被及时通知
  - 希望低成本远程查看，不使用复杂后台

## 3. 产品价值
- 对老人：像“小孙子 / 小孙女”一样陪伴，降低孤独感和使用门槛
- 对家属：通过日志、摘要、通知减少信息盲区
- 对项目交付：保留无第三方 Node 依赖的轻量结构，同时具备升级为正式部署的路径
- 对展示竞争力：避免落入“普通聊天 + 普通提醒”同质化，突出情感与安全双价值

## 5. 产品目标

### 5.1 用户目标
- 老人打开页面后 30 秒内理解如何使用
- 老人能只通过一句自然语言完成提醒创建
- 老人说出高风险表达时，3 秒内看到清晰的求助动作
- 家属能在 10 秒内扫读“最近有没有异常”

### 5.2 业务目标
- 构建一个可运行、可部署、可展示的适老化 Web 产品
- 实现“老人端 + 家属端”的产品闭环
- 提供可继续扩展到正式场景的技术底座

### 5.3 成功指标
- 日均对话次数
- 提醒创建成功率
- 紧急表达识别触发率
- 家属页访问频次
- 通知送达率
- 晨间播报点击或播放频次
- 心情打卡覆盖率
- 时光回忆保存量
- 防诈骗拦截触发量

## 6. 核心使用场景
- 老人觉得闷、孤单，想聊聊天
- 老人需要设定吃药、喝水、复诊提醒
- 老人想问今天天气、适不适合出门散步
- 老人想听新闻、笑话等轻内容
- 老人身体不适，希望知道先做什么
- 老人想简单报个平安，不想写长句
- 家属想给老人留一句暖心留言
- 老人接到可疑电话或转账要求，希望知道该不该信
- 家属希望留住老人说过的故事，而不只是看一堆日志
- 家属想查看近期异常事件和提醒情况
- 家属希望每日收到一条安心摘要

## 8. 核心流程

### 8.1 日常陪伴流程
1. 用户打开首页
2. 系统展示开场白、快捷问题和晨间播报
3. 用户通过语音或文字发起对话
4. 系统按优先级返回回复：
   - 本地知识库
   - 实时服务
   - 防诈骗守护
   - DeepSeek
   - 本地规则兜底
5. 若开启播报，系统自动朗读回复

### 8.2 提醒流程
1. 用户说“每天早上 8 点提醒我吃药”
2. 系统解析时间、频率、标题
3. 生成提醒并写入存储
4. 到点后页面主动播报并记录事件
5. 单次提醒自动停用，每日提醒继续保留

### 8.3 晨间播报流程
1. 首页加载或用户点击刷新
2. 后端聚合天气、正向资讯和提醒摘要
3. 前端展示摘要卡片
4. 用户点击“播放晨间播报”进行语音播报

### 8.4 紧急流程
1. 用户说出高风险表达
2. 系统立即识别并弹出紧急模式
3. 页面明确展示：
   - 先拨打 120
   - 联系紧急联系人
   - 保持电话畅通并说明地址
4. 系统写入高风险事件日志
5. 若配置 webhook，则向家属发送紧急通知

### 8.5 心情报平安流程
1. 老人在首页点击一个心情标签
2. 选择当前精神状态并可补充一句备注
3. 系统保存打卡并更新家属页概览
4. 若状态为“不太舒服”或“有点担心”，可触发家属提示通知

### 8.6 时光记忆流程
1. 首页展示当天的回忆话题
2. 老人通过文字或语音说出一段往事
3. 系统将内容保存到时光回忆册
4. 家属端可查看最近回忆内容，形成情感连接

### 8.7 防诈骗守护流程
1. 老人在对话中提到转账、验证码、公检法、中奖、保健品等关键词
2. 系统优先触发防诈骗守护，而不是普通闲聊回复
3. 页面展示阻断步骤，例如不要转账、不要共享验证码、先联系家人核实
4. 系统记录安全日志，必要时通知家属

### 8.8 家属查看流程
1. 家属进入家属页
2. 查看通知状态、摘要时间、统计卡片
3. 查看最近状态打卡、时光回忆和高风险事件
4. 可发送暖心便签到老人首页
5. 按类型 / 等级 / 关键词筛选日志
6. 可手动发送测试通知或安心摘要

## 9. 功能需求

### 9.1 陪伴对话
- 首页必须展示开场白和预置问题
- 回复语气要简短、亲切、可理解
- 普通闲聊可调用 DeepSeek
- 高风险或提醒类语句不得依赖大模型决定核心动作

### 9.2 本地知识库
- 维护老人常见问答
- 高置信度命中时直接返回本地答案
- 低置信度命中时可作为 Prompt 参考资料

### 9.3 实时天气与正向资讯
- 天气必须支持城市 / 区县级查询
- 若外部源不可用，必须优雅降级
- 资讯以正向、轻量内容为主
- 默认过滤明显负向标题

### 9.4 晨间播报
- 必须包含问候
- 必须包含天气摘要
- 必须包含今日提醒概览
- 必须包含至少一条暖心资讯摘要
- 必须支持前端一键播报

### 9.5 提醒能力
- 支持新增、启用、停用、删除
- 支持“每天”和“仅一次”
- 支持语音和文本两种创建方式
- 到点后必须留痕

### 9.6 紧急识别
- 必须识别胸痛、呼吸困难、摔倒、出血、意识不清、中风、救命等表达
- 只输出求助步骤，不做医疗诊断
- 必须优先展示 120 与联系人

### 9.7 常见症状照护
- 对头晕、头痛、腹痛等高频表达，优先输出简短、同理心强、可执行的照护步骤
- 这类回复不能只剩下“立刻打120”，需要先给出当前可做的安全动作
- 必须写入日志，并支持联动联系人通知
- 不能伪装成医生诊断，只能给出保守、明确的下一步
### 9.8 家属联动
- 支持展示家属配置状态
- 支持测试通知
- 支持手动发送每日安心摘要
- 紧急事件和手动一键求助要支持 webhook 推送
### 9.9 心情报平安
- 首页必须可一键完成状态打卡
- 至少包含心情与精神状态两个维度
- 最近一次状态必须同步展示在家属页
- 低落、不安、不适状态需要支持后续提醒策略

### 9.10 时光记忆卡
- 系统每天至少生成一个回忆话题
- 老人的回忆内容必须可被持久化保存
- 家属页必须能查看最近保存的回忆
- 设计目标不是普通笔记，而是家庭情感资产沉淀

### 9.11 防诈骗守护
- 必须识别验证码、转账、公检法、中奖、保健品等常见诈骗话术
- 触发时要优先给出阻断步骤，而不是继续闲聊
- 必须写入安全日志
- 应支持通知家属进行二次确认

### 9.12 多事项提醒
- 一句话内包含多个事项时，系统应尽量拆成多条提醒，而不是只保留第一项
- 至少支持相同时间下的多事项拆分
- 创建成功后，回复文案要明确告知“已记住几项提醒”

### 9.13 语音能力
- 浏览器支持时优先使用原生语音能力
- 云端 ASR/TTS 已配置时优先走云端
- 云端异常必须自动回退

### 9.14 适老化与双模式
- 长辈模式使用更暖的视觉风格
- 家属模式更强调信息密度和概览效率
- 必须支持大字模式
- 必须支持减少动画

## 12. 非功能需求
- 移动优先
- 支持主流手机浏览器访问
- Node.js 单进程可运行
- 默认无第三方 Node 依赖
- 存储层支持 JSON 与 SQLite 两种模式
- JSON 写入需要原子化
- 可通过 `systemd` 守护
- 可通过 `Nginx` 反向代理
- 麦克风相关能力在正式环境建议运行于 HTTPS

## 13. 安全与合规
- 不提供医疗诊断
- 高风险内容优先引导求助
- 仅保存最小必要个人信息
- 建议正式环境使用 HTTPS
- 建议后续补充登录鉴权和角色控制

## 14. 当前版本验收标准
- 手机浏览器可访问并正常切换页面
- 开场白、预置问题、知识库内容与前端一致
- 首页能看到心情报平安、暖心便签和时光记忆卡
- 自然语言提醒创建可用
- 防诈骗表达能触发守护流程和安全日志
- 紧急表达能触发高风险流程
- 晨间播报接口可返回摘要
- 家属通知测试接口可返回状态
- 心情打卡、时光回忆、暖心便签可持久化
- JSON 模式下可正常运行
- SQLite 模式下可正常运行
- 提供完整 README、部署文档和 PRD
- GitHub 仓库已同步最新代码

## 15. 已知边界与下一步

### 当前边界
- 没有正式登录鉴权
- 没有真实短信 / 电话外呼
- 新闻和天气依赖外部服务，可降级但不保证始终实时

### 建议下一步
1. 增加家属访问码或正式鉴权
2. 接入真实短信 / 电话网关
3. 增加数据备份、监控和审计
4. 加入更细的照护评估与长期记忆策略
