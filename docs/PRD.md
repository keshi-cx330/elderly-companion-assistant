# 老人陪伴助手 PRD（MVP）

## 1. 项目概述
- 产品名称：老人陪伴助手
- 产品形态：手机 Web 应用（浏览器访问）
- 核心交互：语音为主，文字为辅
- 核心定位：
  1. 提供日常陪伴聊天与简单问答。
  2. 提供可执行的提醒能力（吃药、喝水、复诊等）。
  3. 识别高风险表达并进入紧急模式，优先引导求助。
  4. 保留关键记录，支持家属/后台追溯。

## 2. 背景与目标
### 2.1 目标用户
- 主要用户：60+ 老年人（独居或半独居）
- 次要用户：家属/照护者（查看关键记录）

### 2.2 主要痛点
- 陪伴不足：缺少稳定沟通对象。
- 记忆与执行力下降：日常事项容易遗忘。
- 信息获取门槛高：传统搜索流程复杂、字体小。
- 紧急应对弱：突发不适时不清楚如何快速求助。

### 2.3 产品目标（MVP）
- 对话可用：支持语音输入与连续多轮聊天。
- 提醒可用：支持新增/停用/删除提醒并定时触发。
- 紧急可用：识别风险词并弹出明确求助路径。
- 记录可查：对话与事件可查看、可筛选。
- 合规可控：不做医疗诊断，风险内容优先转求助。

## 3. 范围定义
### 3.1 本期范围（In Scope）
- 语音识别（浏览器 Web Speech API）+ 文字输入
- 陪伴聊天（规则引导 + 安全提示）
- 提醒管理（每天/一次）
- 个人资料与紧急联系人管理
- 紧急模式弹窗与求助动作（拨打 120 / 紧急联系人）
- 事件与对话日志（家属查看）

### 3.2 非本期范围（Out of Scope）
- 医疗诊断与病情判断
- 视频通话、复杂 IM、第三方账号体系
- 医院系统打通、真实短信网关集成
- 多语言与多端同步登录

## 4. 用户流程
### 4.1 日常陪伴流程
1. 用户进入首页，默认看到“对话”面板。
2. 用户点击语音按钮说话，或直接输入文字。
3. 系统返回简短、清晰、可执行的回复并可语音播报。
4. 若用户提出提醒诉求，跳转提醒面板进行设置。

### 4.2 提醒设置流程
1. 用户填写提醒内容、时间、频率（每天/一次）。
2. 系统保存提醒并记录“提醒创建事件”。
3. 到点后前端触发提醒播报，并上报“提醒触发事件”。
4. 用户可随时停用/删除提醒。

### 4.3 紧急求助流程
1. 用户输入/说出高风险表达（如“胸痛”“救命”）。
2. 系统识别后进入“紧急模式”弹窗。
3. 页面展示三步操作：
   1. 拨打 120；
   2. 联系紧急联系人；
   3. 保持通话等待救援。
4. 系统记录紧急事件，供家属端追溯。

### 4.4 家属查看流程
1. 进入“家属”面板。
2. 查看概览指标（对话次数、紧急次数、提醒事件等）。
3. 筛选日志类型（全部/对话/提醒/紧急）。

## 5. 交互方式与设计原则
## 5.1 交互原则
- 大按钮、短文本、明确动作。
- 语音优先：按钮一键触发语音识别。
- 关键信息可视化：时间、事件类型、风险等级突出显示。

## 5.2 可用性要求
- 移动优先布局，支持单手点击。
- 对比度充足，字体 >= 14px。
- 错误提示简明，避免复杂术语。

## 6. 核心页面设计
### 6.1 对话页
- 内容：消息气泡、语音按钮、文字输入框。
- 动作：发送文字、开始/停止语音。
- 异常：语音能力不可用时自动降级为文字。

### 6.2 提醒页
- 内容：新增提醒表单、提醒列表。
- 动作：新增、启用/停用、删除。

### 6.3 我的页
- 内容：称呼、年龄、偏好、备注、紧急联系人。
- 动作：保存资料（用于个性化与紧急联系）。

### 6.4 家属页
- 内容：统计卡片、日志筛选、事件列表。
- 动作：刷新、按类型筛选。

### 6.5 紧急弹窗
- 内容：高风险提示、三步求助流程、电话按钮。
- 动作：拨打 120、联系紧急联系人、关闭弹窗。

## 7. 数据与接口
### 7.1 数据实体
- `profile`：用户基本信息与紧急联系人
- `reminder`：提醒标题、时间、频率、状态
- `conversation`：用户/助手对话记录
- `event`：提醒、紧急、资料变更等事件

### 7.2 核心 API（已实现）
- `GET /api/health`
- `GET /api/profile` / `PUT /api/profile`
- `GET /api/reminders` / `POST /api/reminders`
- `PATCH /api/reminders/:id` / `DELETE /api/reminders/:id`
- `POST /api/reminders/trigger`
- `POST /api/chat`
- `POST /api/emergency/report`
- `GET /api/dashboard`
- `GET /api/logs?type=...`

## 8. 非功能需求
- 部署：可部署在任意 Linux/Windows 服务器，端口可配置。
- 性能：单实例满足小规模家庭场景。
- 安全：仅收集最小必要信息，不提供医疗诊断建议。
- 可维护性：无外部依赖，后端与前端代码分离。

## 9. 三天开发计划
### Day 1
- 完成需求梳理、信息架构、API 设计、后端数据模型。

### Day 2
- 完成移动端 UI、语音交互、提醒与紧急流程联调。

### Day 3
- 完成日志追溯、文档完善、部署验证、验收演示。

## 10. 验收标准
- 手机浏览器可访问并完成完整链路：
  1. 语音/文字对话；
  2. 新增并触发提醒；
  3. 识别紧急表达并展示求助动作；
  4. 家属页可查看日志与统计。
- 提交完整 PRD 与部署文档。
- Git 保留完整提交记录。
