# 老人陪伴助手 PRD

## 1. 产品概述
- 产品名称：老人陪伴助手
- 产品形态：手机 Web 应用，浏览器访问
- 核心交互：语音为主，文字为辅
- 核心定位：帮助独居或半独居老人以最低学习成本完成陪伴聊天、提醒管理与紧急求助
- 产品愿景：让老人“开口就能用”，在日常生活中像晚辈一样陪伴，在关键时刻像应急助手一样给出明确行动

## 2. 用户与场景

### 2.1 目标用户
- 主要用户：60 岁以上老年人，尤其是独居、空巢、慢病管理人群
- 次要用户：家属、照护者、社区工作人员

### 2.2 典型使用场景
- 日常无聊时，希望有人聊天
- 吃药、喝水、复诊、出门买菜等容易遗忘
- 身体不适时不知道下一步怎么办
- 家属希望查看老人近期异常记录、提醒执行情况

### 2.3 主要痛点
- 孤独感重，但不愿意使用复杂社交软件
- 手机字体小、层级深、操作复杂
- 不会精准搜索，不愿意打字
- 突发风险时容易慌乱，无法迅速做出求助动作

## 3. 产品目标

### 3.1 业务目标
- 三天内交付一个可部署到 Linux 服务器、可被手机浏览器访问的产品原型
- 实现完整闭环：进入页面 -> 对话/提醒 -> 紧急求助 -> 家属追溯

### 3.2 用户目标
- 老人 30 秒内理解如何使用
- 老人可以仅通过语音完成至少一条提醒创建
- 遇到高风险表达时能在 3 秒内看到紧急求助步骤

### 3.3 成功指标
- 日均发起对话次数
- 提醒创建成功率
- 高风险表达识别触发率
- 家属查看日志频次
- 页面首屏可交互时间

## 4. 范围定义

### 4.1 In Scope
- 浏览器语音识别与语音播报
- 陪伴聊天
- 文字输入兜底
- 自然语言创建提醒
- 手动提醒管理
- 紧急联系人管理
- 高风险紧急求助弹窗
- 家属概览与日志查看
- Linux 部署、运行说明、PWA 外壳

### 4.2 Out of Scope
- 医疗诊断、处方建议
- 视频通话、IM 即时消息系统
- 真实短信、电话网关自动外呼
- 多账号、多租户、权限系统
- 第三方健康平台数据打通

## 5. 核心价值主张
- 对老人：不需要学习复杂 App，只要会说话就能用
- 对家属：能看到紧急记录、提醒与关键事件，不再完全依赖口头转述
- 对交付：无第三方依赖，部署简单，适合比赛、课程或快速验收项目

## 6. 角色画像

### 6.1 老人用户
- 年龄：60-80 岁
- 设备：智能手机
- 偏好：大字、大按钮、语音操作
- 风险：紧急情况下表达不完整、手抖、视力弱、容易慌

### 6.2 家属用户
- 关注点：老人今天是否互动过、提醒是否触发过、最近是否有异常
- 使用习惯：偶尔打开页面快速扫一眼，不希望进入复杂后台

## 7. 用户故事
- 作为老人，我希望打开页面就能看到一个明显的语音按钮，这样我不用先学习怎么操作
- 作为老人，我希望直接说“每天早上 8 点提醒我吃药”，系统就能替我创建提醒
- 作为老人，当我说“胸口很痛”时，系统应该立刻告诉我先做什么
- 作为家属，我希望看到最近 7 天有没有紧急事件
- 作为家属，我希望通过关键字搜索“摔倒”或“吃药”快速找到相关记录

## 8. 信息架构
- 对话页：核心入口，语音按钮、文字输入、快捷语句
- 提醒页：今日安排、提醒表单、提醒列表
- 我的页：老人资料、地址、联系人、可访问性设置
- 家属页：统计概览、风险提示、筛选日志
- 紧急弹窗：三步求助动作、120、联系人按钮

## 9. 关键流程

### 9.1 日常陪伴流程
1. 用户进入首页
2. 默认落在“对话”页
3. 用户点语音按钮说话，或通过大字文本框输入
4. 系统返回简短回复，并按设置进行语音播报
5. 若识别到提醒意图，直接创建提醒并给出确认

### 9.2 提醒流程
1. 用户手动进入提醒页，或通过语音直接创建提醒
2. 系统保存提醒，并记录创建事件
3. 到点后页面主动播报提醒并触发记录
4. 单次提醒自动停用；每日提醒保留

### 9.3 紧急流程
1. 用户说出“胸痛”“摔倒”“救命”等高风险表达
2. 系统识别后立即弹出紧急模式
3. 页面展示明确步骤：
   - 立即拨打 120
   - 联系紧急联系人
   - 保持电话畅通并说明地址
4. 系统记录事件，供家属追溯

### 9.4 家属查看流程
1. 家属打开家属页
2. 查看统计卡片、最近高风险事件
3. 通过类型、等级、关键词筛选日志

## 10. 核心页面设计要求

### 10.1 对话页
- 必须有明显语音入口
- 必须支持文字输入兜底
- 必须展示快捷操作芯片
- 对话气泡字号大于普通聊天软件默认字号

### 10.2 提醒页
- 今日安排需要突出展示
- 新增提醒表单字段必须简单
- 支持“每天”和“仅一次”

### 10.3 我的页
- 信息最小化，只保留必要字段
- 地址必须可填写，方便紧急求助
- 设置项必须可视化，不隐藏在二级页面

### 10.4 家属页
- 概览指标必须可扫读
- 高风险信息必须优先展示
- 日志列表必须可筛选

### 10.5 紧急弹窗
- 颜色、文案、按钮必须明显区分于普通状态
- 按钮顺序默认以 120 为第一优先级

## 11. 功能需求

### 11.1 陪伴对话
- 支持问候、聊天、提醒引导、联系家人、感谢等基础意图
- 回复不应过长，避免给老人造成认知负担

### 11.2 紧急识别
- 识别胸痛、呼吸困难、摔倒、出血、昏迷、救命等关键词
- 不输出诊断结果
- 只做求助引导

### 11.3 提醒能力
- 手动新增提醒
- 自然语言解析提醒
- 提醒启用/停用/删除
- 提醒触发自动留痕

### 11.4 数据留痕
- 对话记录
- 提醒创建/触发/更新/删除
- 紧急识别与紧急上报
- 资料和设置更新

- `GET /api/profile`

- `GET /api/settings`
- `PUT /api/settings`
- `GET /api/reminders`

- `PATCH /api/reminders/:id`
- `DELETE /api/reminders/:id`
- `POST /api/reminders/trigger`
- `POST /api/chat`
- `POST /api/emergency/report`
- `GET /api/dashboard`
- `GET /api/logs`

## 14. 非功能需求
- 移动优先布局
- 兼容主流手机浏览器
- Node.js 单进程即可运行
- 无第三方运行时依赖
- JSON 文件原子写入，减少数据损坏风险
- 可在 Linux 服务器上通过 `systemd` 守护
- 可被 `Nginx` 代理

## 15. 安全与合规
- 不提供医疗诊断
- 高风险内容优先引导求助
- 仅保存最小必要个人信息
- 建议正式环境启用 HTTPS

## 16. 验收标准
- 手机浏览器可访问
- 语音识别不可用时文字输入仍可完整使用
- 能通过自然语言创建提醒
- 能识别高风险表达并弹出紧急模式
- 家属页能看到统计和日志
- 提供完整 PRD、README、Linux 部署文档
- 项目保留 Git 提交记录
